# CLAUDE.md — Airport Ground Handling Optimizer

## Launch
```bash
python src/pipeline.py --config configs/scenario_1.yaml --seed 42
```
Одна команда, без ручных правок путей. Чистая Ubuntu 20/22/24 + Python 3.10+.

## Stack
- Python 3.11
- scikit-learn (RandomForestRegressor)
- NetworkX + rule-based greedy scheduler
- matplotlib (графики сохраняются как .png)
- YAML (конфиги)

## Модули — строго раздельные
| Файл | Отвечает за |
|------|-------------|
| `src/data_generator.py` | Синтетические данные |
| `src/ml_model.py`       | Прогноз времени обслуживания |
| `src/optimizer.py`      | Назначение ТС, маршруты |
| `src/simulator.py`      | Симуляция выполнения |
| `src/metrics.py`        | KPI, графики |
| `src/pipeline.py`       | Точка входа |

Forecast ≠ Optimization ≠ Simulation ≠ Metrics. Не смешивать логику между модулями.

## Data generator
- Обязателен `seed=42` — полная воспроизводимость
- Генерирует: расписание рейсов, запросы на обслуживание, геометрию перрона, парк ТС
- Реальные данные не нужны — только синтетика

## Авиационные ограничения (жёсткие)
- Приоритет: деайсинг > заправка > кейтеринг
- Временные окна STD/STA соблюдаются строго
- Совместимость ТС по типу и вместимости
- Безопасные интервалы между операциями
- Скоростные ограничения на перроне

## Конфигурируемость
Все параметры выносятся в `configs/scenario_1.yaml`:
- Веса приоритетов
- Пороги ограничений
- Параметры парка ТС
- Параметры сценария

Никаких хардкоженных чисел в коде.

## Обработка краевых случаев
Каждый модуль обязан обрабатывать:
- Пустой вход
- Неразрешимые ограничения (нет подходящего ТС)
- Падение ML-модели (fallback на среднее значение)
- Конфликты временных окон

## Тесты — минимум 3
1. Валидация авиационных ограничений (приоритеты, временные окна)
2. Корректность расчёта метрик
3. Детерминированность при `seed=42` — два запуска дают идентичный результат

## Порядок работы над каждым модулем

```
1. Код        — реализовать модуль согласно module_interfaces.md
2. Тест       — запустить, убедиться что работает без ошибок
3. Багфикс    — если есть ошибки, исправить до перехода дальше
4. Коммит     — зафиксировать рабочее состояние в git
5. Документация — добавить docstring к функциям модуля
```

Не переходить к следующему модулю пока текущий не прошёл все 5 шагов.
