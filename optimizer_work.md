# Optimizer — логика работы

## Входные данные
- `tasks_df` — список задач обслуживания (от DataGenerator)
- `vehicles_df` — парк ТС (тип, вместимость, текущая позиция)
- `apron_graph` — граф перрона (NetworkX)
- `tasks_df['service_time_pred']` — прогноз времени обслуживания от MLForecast (мин)

## Шаг 1 — Вычисление urgency (влияние ML)

```
urgency = STD - earliest_start - service_time_pred
```

- `STD` — запланированное время вылета рейса
- `earliest_start` — минимальное время начала задачи (STA + буфер постановки ВС)
- `service_time_pred` — выход ML-модели (мин)

Чем меньше urgency → тем задача срочнее.

## Шаг 2 — Сортировка задач

Двухуровневая сортировка перед назначением:

1. `priority_group` по возрастанию (деайсинг=1 > заправка=2 > кейтеринг=3)
2. `urgency` по возрастанию внутри группы

Жёсткий приоритет группы не нарушается urgency.

## Шаг 3 — Greedy назначение ТС

Для каждой задачи в отсортированной очереди:

1. Найти ТС подходящего типа, который свободен раньше всех
2. Рассчитать маршрут до задачи через `apron_graph` (NetworkX shortest path)
3. Назначить ТС, обновить его время освобождения: `free_at = start + service_time_pred`
4. Если подходящего ТС нет → зафиксировать нарушение (violation), пропустить задачу

## Выход

- `assigned_routes` — список: {task_id, vehicle_id, start_time, end_time, route}
- `violations` — задачи без назначенного ТС или с нарушением временного окна
